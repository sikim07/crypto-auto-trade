# crypto-auto-trade

Upbit KRW 마켓 대상 암호화폐 자동매매 봇. 대시보드 웹앱에서 시작하여, 1분봉 스캘핑 전략(A~F)을 거쳐, 그리드 트레이딩으로 전환한 프로젝트.

이전 레포: [upbit-trade-bot](https://github.com/sikim07/upbit-trade-bot) (2025.10 ~ 2026.02)

## 프로젝트 개요

- 기간: 2025.10 ~ 2026.06 (약 9개월)
- 언어: TypeScript (Node.js)
- 거래소: Upbit (한국 원화 마켓)
- 운영: PM2 프로세스 매니저

## 구조

```
src/
├── common/           # 공통 모듈 (타입, 로거, 환경변수)
├── upbit/            # Upbit API 래퍼 (REST, WebSocket, JWT 인증)
├── grid/             # 그리드 트레이딩 봇 (최종 버전)
│   ├── gridBot.ts    # 메인 루프 (초기화 → 폴링 → 종료)
│   ├── gridConfig.ts # 그리드 설정값
│   ├── gridState.ts  # 상태 관리 + JSON 백업/복구
│   ├── gridOrders.ts # 지정가 주문 배치/체결 감지
│   ├── trendGuard.ts # 추세 감지 → 자동 중단/재개
│   └── gridReport.ts # 운영 현황 리포트
└── _legacy/          # 폐기된 스캘핑 전략 코드 (빌드 제외)
```

## 전략 변천사

### 0단계: 대시보드 + 시뮬레이션 (2025.10 ~ 2026.02)

코인 시세 모니터링 대시보드와 매매 시뮬레이션 기능을 만들었다. Express + EJS 기반 웹앱으로 시작해서, NestJS + React로 전환을 시도했다가, 다시 Express로 롤백한 뒤, 최종적으로 대시보드를 버리고 매매 봇만 분리했다.

### 1단계: 1분봉 스캘핑 (2026.02 ~ 2026.06)

Upbit KRW 마켓에서 1분봉 기술적 지표 기반으로 단타 매매를 시도했다. 전략 A~F까지 6개를 만들고 수십 차례 파라미터를 조정했으나, 수수료 구조를 넘는 엣지를 만들지 못했다.

**실전 결과 (9일간 33건)**

- 승률 27% (9승 24패)
- 평균 수익 +0.61%, 평균 손실 -0.85%
- 기대값 -0.45%/건 (매매할수록 손실)
- 42%의 진입이 한 번도 수익 구간에 진입하지 못함

### 2단계: 그리드 트레이딩 (2026.06 ~)

방향 예측이 필요 없는 전략으로 전환. 일정 가격 범위를 N단계로 나누고 각 단계에 매수/매도 지정가 주문을 배치하여 횡보장에서 가격 왕복으로 수익을 수확하는 구조.

### 3단계: DEX-CEX 차익거래 (시도 후 폐기)

Binance(CEX)와 Jupiter/Solana(DEX) 간 가격 차이를 이용한 차익거래를 구현했으나, 소액으로는 가스비와 슬리피지를 감당할 수 없어 폐기.

### upbit-trade-bot (이전 레포)

#### 2025.10.02~03 -- 프로젝트 시작, CI/CD 구축

EC2에 배포하는 GitHub Actions CI/CD 파이프라인 구성. Upbit API로 거래대금 상위 코인을 선별하여 WebSocket으로 구독하는 프로세스 구현. yml 파일 설정에서 반복적으로 시행착오.

#### 2025.10.07 -- 대시보드 + PM2 구성

변동성 높은 코인을 보여주는 웹 대시보드 추가. PM2로 프로세스 관리 시작. 모바일 PWA 대응.

- 문제: PM2 namespace 설정에서 삽질 (커밋 4개가 PM2 설정 수정).

#### 2025.10.13 -- 대규모 리팩토링

JavaScript에서 TypeScript로 전환. 파일 구조 모듈화. ESLint 적용. 절대경로 전환. 상수 분리. class 형식을 const 형식으로 변경. PWA 파비콘, 다크모드, 계좌 조회, 거래 내역 페이지 등 프론트엔드 기능을 하루만에 대량 추가. 이 날 하루에 커밋 47개.

- 문제: EC2에서 `PROJECT_ROOT` 경로가 빌드 환경과 달라서 정적 파일 서빙이 안 되는 문제. 디버깅 로그를 넣었다 빼는 커밋이 반복됨.
- 배운 것: 경로 관련 로직은 환경별 차이를 고려해서 설계해야 한다. 하루에 기능을 너무 많이 넣으면 문제 원인 추적이 어렵다.

#### 2025.10.14 -- 시뮬레이션 기능 구현

단타 매매 전략 시뮬레이션 페이지 추가. Jest 테스트 구조 구성. 대시보드 초기 데이터 로딩 개선. Socket.IO를 추가했다가 바로 revert.

- 문제: Socket.IO가 서버에서 소켓 신호를 가로채는 현상. Jest가 종료되지 않는 문제.
- 배운 것: 라이브러리를 추가하기 전에 기존 구조와 충돌 여부를 먼저 확인해야 한다.

#### 2025.10.18~19 -- 전략 업데이트, API 디버깅

매매 전략을 추가하고 실전 API를 연동하면서 데이터 조회 오류 반복. 거래내역 API, 계좌 조회 API에서 오류 원인을 찾기 위해 로그 추가/제거를 반복.

- 문제: API 응답 구조를 제대로 파악하지 않고 구현해서, 디버깅에 시간 소모.
- 배운 것: 외부 API 연동 시 응답 스키마를 먼저 정확히 파악하고 타입을 정의해야 한다.

#### 2025.10.20 -- UI/UX 대량 개선

대시보드 가격 변동 시각화, 시뮬레이션 상태 영속화, 모바일 탭 네비게이션, 다크모드 토글, Pull to Refresh 등. 이 날에만 커밋 40개 이상.

- 배운 것: 프론트엔드 작업은 모바일/데스크탑을 동시에 고려해야 하고, !important는 피해야 한다.

업비트 시장환경을 실시간으로 판단하는 시스템 추가. API Rate Limit 처리.

- 배운 것: 외부 API에 대한 rate limiting 처리를 기본으로 구현해야 한다.

#### 2025.11.03~10 -- API 안정화, 서버 이관

상장폐지/거래불가 코인 현재가 조회 실패 처리. 대시보드 API 중복 호출 방지. AWS EC2에서 Vultr로 서버 이관. SSH 설정에서 또 시행착오 (커밋 6개가 SSH 수정).

- 문제: EC2 ubuntu 유저에서 Vultr root 유저로 바뀌면서 CI/CD 전체가 깨짐.
- 배운 것: 서버 이관 시 사용자 권한, 경로 등 환경 차이를 체크리스트로 관리해야 한다.

#### 2025.11.22~12.06 -- NestJS + React 전환 시도

Express + EJS 구조를 NestJS + React + TailwindCSS 모노레포로 전환 시도. Docker 도입. Shadcn UI 적용. 이 기간 동안 커밋 메시지에 "t", "ㅅ" 같은 디버깅 커밋이 다수.

- 문제: 프레임워크 전환 자체가 목적이 되어, 실제 매매 기능 개발이 진행되지 않았다. 배포 오류 수정에 시간 대부분을 소모.
- 배운 것: 동작하는 코드를 프레임워크 전환하는 것은 새로 만드는 것보다 어렵다. 작동하는 상태를 유지하면서 점진적으로 바꿔야 한다.

NestJS 전환을 포기하고 Express로 롤백. 기존 미사용 파일 정리.

- 배운 것: 2개월간의 프레임워크 전환 작업이 전부 버려졌다. 기술 스택 변경은 명확한 이유와 단계적 계획이 없으면 시간 낭비.

#### 2026.01.19~27 -- Bulma UI 전환, 기능 추가

Bulma CSS 프레임워크 적용. 로그인 기능. 시뮬레이션 UI 개선. 각 페이지 모바일/데스크탑 UI 정비.

- 문제: UI 프레임워크를 또 바꿈 (기존 커스텀 CSS → TailwindCSS → Shadcn → Bulma). 매번 전체 스타일을 다시 작업.
- 배운 것: UI 프레임워크는 초반에 하나를 정하고 끝까지 가야 한다. 중간에 바꾸면 기존 작업이 전부 손실.

#### 2026.02.04~06 -- 실거래 기능 추가

시뮬레이션이 아닌 실제 매매 기능 추가. 매매 전략 3종 구현. 대시보드에서 실거래 시작/중단 제어.

- 문제: 대시보드 + 실거래를 한 프로젝트에서 관리하다 보니 코드가 복잡해짐. 프론트엔드 코드와 매매 로직이 뒤섞임.
- 배운 것: 대시보드(시각화)와 매매 봇(실행)은 분리해야 한다. 이 판단이 이후 새 레포(crypto-auto-trade) 생성으로 이어짐.

### crypto-auto-trade (현재 레포)

#### 2026.02.26 -- 매매 봇 분리, 첫 배포

대시보드 없이 매매 봇만 별도 레포로 분리. 단일 전략으로 시작. 매수/매도 조건 완화, 동시 매수 플래그 충돌, 매수 후 volume이 0으로 조회되는 등 기본적인 방어 로직이 빠져 있었다.

- 배운 것: 거래소 API는 예상 외의 응답을 자주 준다. 모든 API 응답에 방어 로직이 필요하다.

#### 2026.02.28 -- 매수 조건 디버깅

거래량 계산 오류 수정, 매수 기회가 1시간 없으면 종목을 재선정하는 로직 추가. 로그를 세분화하면서 문제 원인을 추적하기 시작했다.

- 배운 것: 로그가 없으면 실전에서 아무것도 분석할 수 없다. 초기부터 체계적인 로깅이 필요하다.

#### 2026.03.03 -- 전략 세분화 (A~E)

하나의 전략으로는 다양한 시장 상황에 대응할 수 없다고 판단하여, 시장 상황별 전략을 분리했다. JWT 인증 오류(query string 해시 방식) 수정. 손익 로그 추가.

- 배운 것: Upbit JWT 인증에서 query parameter의 해시 생성 방식이 문서와 미묘하게 다른 부분이 있다 (states[] → state 변환 등).

#### 2026.03.04~06 -- 시장 레짐 필터 도입

하락장에서 매수를 방지하는 레짐 필터 추가. 전략 F(VWAP 눌림목 반등) 개발 시작. 전략별 백테스트 로그 추가.

- 배운 것: 아무리 좋은 진입 조건도 시장 전체가 하락 중이면 무의미하다. 시장 상황 판단이 개별 전략보다 먼저다.

#### 2026.03.08~12 -- 반복적인 전략 튜닝

"시장 진입 조건 강화", "매수 필터 완화", "전략 3차/4차 개선" 등 파라미터 조정이 반복되었다.

- 배운 것: 파라미터를 건드릴 때마다 한쪽이 좋아지면 다른 쪽이 나빠지는 패턴. 근본적인 엣지가 없으면 튜닝으로는 해결 안 된다.

#### 2026.03.15~25 -- 전략 우선순위 실험

승률이 높은 전략 F부터 먼저 매수 조건을 확인하도록 수정. 전략 B(손실 최대)를 후순위로 밀고, 전략 D 비활성화.

- 배운 것: 6개 전략을 동시에 운영하면 전략 간 상호작용을 통제할 수 없다. 가장 잘 되는 전략 하나에 집중하는 것이 낫다.

#### 2026.04.02~09 -- 1시간봉 전환 시도

1분봉 노이즈가 너무 심해서 1시간봉으로 전환을 시도했으나(T1 전략), 실전에서 기회가 너무 적어 폐기하고 다시 단타로 전환.

- 배운 것: 타임프레임을 바꿔도 수수료 대비 수익 구조가 개선되지 않으면 동일한 문제가 발생한다.

#### 2026.04.14~21 -- 전략 축소와 복원

전략 A 비활성화, F만 남김. 여러 차례 revert 발생 (d0f116c 시점으로 복원, afb4e0e 시점으로 복원). 로그 시스템 오류 수정.

- 배운 것: 실험적 변경은 반드시 브랜치를 분리해야 한다. main에서 직접 실험하면 안정적인 복원 지점을 관리하기 어렵다.

#### 2026.05.04~06 -- 전략 F 진단

매매가 수행되지 않는 이유를 파악하기 위해 진단 로그를 대량 추가. 필터 조건이 너무 까다로워 실제 진입 기회가 거의 없었던 것이 원인.

- 배운 것: 필터를 추가할수록 백테스트 승률은 오르지만, 실전 매매 횟수가 0에 수렴한다. 과최적화(overfitting)의 전형.

#### 2026.05.25~06.11 -- 마지막 스캘핑 시도

전략 F의 필터를 완화하여 실전 투입. 9일간 33건 매매 결과, 기대값 -0.45%/건으로 확정적 손실 구조 확인.

- 배운 것: Upbit 수수료 0.25%(왕복 0.5%)는 1분봉 스캘핑에서 극복하기 어려운 구조적 비용이다. 목표 수익 0.5%의 절반이 수수료로 사라진다.

#### 2026.06.17 -- 그리드 트레이딩 전환

기존 스캘핑 코드를 `_legacy/`로 이동하고, 그리드 트레이딩 봇을 새로 구현. DEX-CEX 차익거래 봇도 동시에 구현했으나, 소액으로는 실효성이 없어 바로 폐기.

- 배운 것: 방향 예측이 필요 없는 전략이 개인 트레이더에게 적합하다. 단, 그리드도 추세장에서는 한쪽 주문만 체결되어 손실이 누적되므로 추세 감지(trendGuard)가 필수.

#### 2026.06.18 -- 그리드 버그 수정, 프로젝트 정리

그리드 구현 오류 수정. 프로젝트를 포트폴리오로 정리.

## 왜 실패했는가

### 1. 수수료 구조의 한계

Upbit 수수료는 매수 0.05% + 매도 0.05% = 왕복 0.1%. 여기에 호가 스프레드까지 고려하면 실질 비용은 0.2~0.3%에 달했다. 1분봉 스캘핑에서 목표 수익이 0.5%인데 비용이 절반을 차지하면, 승률 60% 이상이 아니면 수익이 불가능하다. 실전 승률은 27%였다.

### 2. 과최적화(Overfitting)

전략 A~F를 만들고 수십 차례 파라미터를 조정하면서, 과거 데이터에서는 잘 작동하지만 실전에서는 통하지 않는 전략이 계속 만들어졌다. 필터를 추가할수록 백테스트 승률은 올라가지만 실전 매매 횟수는 0에 수렴하는 과최적화의 전형적인 패턴.

### 3. 구조적 비대칭

평균 수익(+0.61%)보다 평균 손실(-0.85%)이 컸다. 손절 기준을 넓게 잡으면 한 번의 손실이 여러 번의 수익을 상쇄하고, 손절을 타이트하게 잡으면 노이즈에 걸려 승률이 급락한다. 이 딜레마를 1분봉 단위에서 해결하지 못했다.

### 4. 로그/분석 체계의 부재 (초기)

초기에는 로그 체계 없이 시작해서, 왜 진입했는지/왜 손절됐는지 사후 분석이 불가능했다. 중반 이후 로그를 정비했지만, 이미 많은 시간을 소모한 뒤였다.

### 5. 프레임워크/UI 변경에 시간 소모

이전 레포에서 Express → NestJS → Express 롤백, CSS → TailwindCSS → Shadcn → Bulma 전환을 반복하면서 약 2개월을 프레임워크 전환에 소모했다. 실제 매매 전략 개발과 무관한 작업이었다.

## 다음에 고려할 것

1. **전략 검증을 실전 전에 끝내기**: 수수료 포함 시뮬레이션을 충분히 돌린 뒤 실전 투입. 백테스트에서 기대값이 양수가 아니면 실전에 넣지 않는다.
2. **수수료가 낮은 환경 선택**: 해외 거래소(Binance: maker 0.02%)나 DEX를 사용하면 수수료 부담이 크게 줄어든다.
3. **방향 예측 의존도 줄이기**: 그리드, 마켓메이킹, 차익거래 등 방향 중립 전략이 개인에게 더 적합하다.
4. **실험은 브랜치에서**: main에서 직접 파라미터 실험을 하면 안정적인 복원 지점 관리가 어렵다.
5. **로그 체계를 먼저 만들기**: 매매 로그, 진단 로그, 시스템 로그를 분리하고 분석 파이프라인을 먼저 구축한 뒤 전략을 개발한다.
6. **기술 스택은 초반에 정하고 유지하기**: 프레임워크/UI 라이브러리를 중간에 바꾸면 기존 작업이 전부 손실된다.

## 실행 방법

```bash
# 의존성 설치
npm install

# 환경변수 설정
cp .env.example .env
# .env에 Upbit API 키 입력

# 빌드
npm run build

# 실행
npm start
# 또는 PM2로 운영
pm2 start ecosystem.config.js
```

## 기술 스택

- TypeScript / Node.js
- Upbit Open API (REST + WebSocket)
- PM2 (프로세스 관리)
- axios, ws, dotenv
